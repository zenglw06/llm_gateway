# 安全策略

## 支持的版本

我们会为以下版本提供安全更新：

| 版本 | 支持状态 |
| ---- | -------- |
| 1.x  | ✅ 支持  |
| <1.0 | ❌ 不支持 |

## 报告安全漏洞

我们非常重视项目的安全性。如果您发现了安全漏洞，请按照以下方式报告：

### 报告方式
- 发送邮件到：[security@example.com](mailto:security@example.com)
- 请不要在公开的GitHub Issue中报告安全漏洞

### 报告内容
请在报告中包含以下信息：
- 漏洞的类型和影响范围
- 复现漏洞的步骤（请提供尽可能详细的信息）
- 潜在的影响和风险
- 任何可能的修复建议（可选）

### 处理流程
1. 我们会在24小时内确认收到您的报告
2. 我们会对漏洞进行评估和验证，通常在3个工作日内给出初步反馈
3. 我们会尽快开发修复方案，并与您沟通修复进展
4. 在漏洞修复发布后，我们会在Release Note中说明，并感谢您的贡献

### 公开披露政策
- 我们会在漏洞修复发布后的7天内公开披露漏洞详情
- 我们会与您协调披露的时间和内容，确保不会对用户造成不必要的风险
- 我们会在安全公告中注明报告者的姓名/组织（如果您同意的话）

## 安全最佳实践

### 部署建议
- 始终使用最新的稳定版本
- 不要将管理API暴露在公网
- 配置合理的API Key权限和IP白名单
- 定期审计API Key使用情况和访问日志

### 配置建议
- 使用强密码和随机JWT密钥
- 不要在配置文件中明文存储敏感信息，建议使用环境变量或密钥管理服务
- 合理配置限流策略，防止恶意攻击和滥用
- 开启请求日志，便于安全审计和问题排查

### 数据安全
- 不要将请求/响应日志中的敏感信息（如API Key、用户隐私数据）持久化存储
- 定期清理过期的日志和数据
- 数据传输使用HTTPS加密
